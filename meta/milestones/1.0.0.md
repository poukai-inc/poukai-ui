# @poukai-inc/ui — 1.0.0 milestone

**Author**: poukai-design
**Last updated**: 2026-05-20
**Status**: Shipped 2026-05-20

---

## 1. Why 1.0.0

The package is on `0.22.0`. Thirty-one components are shipped across three atomic layers, covering every surface the pouk.ai site rebuild needs through Phase 4. The token contract is stable — the palette, type scale, spacing ramp, and motion catalog have not changed shape in several releases; only additive tokens have landed. The auto-merge pipeline, Ladle showcase, and three-browser CT suite are all running. Two consumers are in view: the public site and a nascent internal product surface.

The case for 1.0.0 is not "all features done." It is "the contract is stable enough that a rename or removal would be a meaningful breakage for a real consumer." That threshold is met. The case against shipping immediately is that two spec files are missing, one open prop-naming question exists in Wordmark stories, and the CONTRIBUTING guide does not yet exist. These are weeks of effort, not months. The freeze should be planned now so gate items land deliberately rather than after an accidental breaking change forces the version bump.

The case for waiting longer is weak. Every additional `0.x` release that introduces a prop rename or token removal before the freeze happens anyway as a silent breakage. Formalising the contract now costs one planning cycle. Deferring it costs consumer trust on the first breakage.

## 2. Scope of the freeze

The following are **frozen at 1.0.0**. Any change to them after tagging is a major version bump.

**Component prop signatures.** Every named prop on every exported component. This includes prop names, accepted value types (union string literals, boolean flags, ReactNode slots), and required vs. optional status. Adding a new optional prop is minor. Removing or renaming any existing prop is major.

**Token names.** Every CSS custom property defined in `src/tokens/tokens.css`. Renaming or removing a token is major. Adding a new token is minor (consumers ignore unknown custom properties). Changing a token's _value_ without changing its name is a patch if the visual delta is imperceptible, minor if it is intentional and visible.

**Subpath exports.** The five public subpaths: `@poukai-inc/ui`, `@poukai-inc/ui/atoms`, `@poukai-inc/ui/molecules`, `@poukai-inc/ui/organisms`, `@poukai-inc/ui/tokens`, `@poukai-inc/ui/llms.txt`, `@poukai-inc/ui/llms-full.txt`. Adding a new subpath is minor. Removing or renaming one is major.

**CSS class hashing strategy.** The package ships CSS Modules with Vite's default content-hash class names. That strategy is frozen — consumers do not depend on the hash values (they are not public API), but if the hashing algorithm changes the generated CSS would diff, potentially breaking snapshot tests in consuming repos. Any change to the hash strategy requires a major bump and a migration note.

The following are **not frozen** — they can change in minor or patch releases:

- CSS visual output for existing tokens (color value tweaks, animation curve adjustments) — these are patch-level visual refinements, not API breaks.
- Internal component implementation (DOM structure, intermediate class names, ref forwarding internals not exposed as props).
- New components — always minor, never major.
- Story files, Ladle showcase, CI configuration — not public API.
- `meta/` documentation — never versioned.

## 4. Migration guide template

Each future major bump that affects consumers gets a guide at `meta/migrations/<version>.md`. The template structure is:

```
# Migration guide: X.0.0

**Breaking changes**: N
**Estimated upgrade time**: [minutes | hours]

---

## What broke

A table or list of every removed or renamed prop/token/export.

| Surface      | Before         | After         | Reason          |
| ------------ | -------------- | ------------- | --------------- |
| ...          | ...            | ...           | ...             |

## How to upgrade

Step-by-step. Mechanical where possible; codemod where practical.

## Why we changed

The design rationale. Why the old shape was wrong and the new one is better. This is not an apology — it is the documentation that prevents the same mistake in two years.
```

### Worked example: hypothetical `Field.label` → `Field.title` rename

**What broke**

| Surface   | Before       | After        | Reason                                                                                                                                                  |
| --------- | ------------ | ------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `<Field>` | `label` prop | `title` prop | `label` conflicted with the native HTML `<label>` element's text; `title` is the visible heading above the field, distinct from the programmatic label. |

**How to upgrade**

Find every `<Field label=` in your codebase and replace with `<Field title=`. The rendered output is identical. If you are using TypeScript, the compiler will surface every remaining instance after the package update.

```bash
# Quick find — verify before bulk-replacing
grep -r "Field label=" src/
```

**Why we changed**

`Field.label` was ambiguous: it described the visible text above the field but shared its name with the `<label>` HTML element's accessible text content. Consumers reading the prop table could not tell whether `label` referred to the `<label>` element or the visible heading. `title` is unambiguous — it is the displayed heading. The `<label>` element's `for`/`htmlFor` association remains internal to `Field` and is not a consumer-facing prop.

## 5. Out of scope for 1.0

The following are explicitly deferred. They are not omissions — they are documented post-1.0 candidates to anchor scope and prevent scope creep into the freeze window.

**Toast queue persistence.** The current `Toast` organism auto-dismisses and does not survive page navigation in SPAs. Persisting the queue across route changes requires either a top-level context outside the React tree (e.g. a Zustand store) or a native browser API integration. Neither is appropriate for a DS primitive at this stage — the consuming surface has not asked for it, and the right architecture depends on what router and state management the consumer owns.

**Form validation hook contract.** `Form` is shipped as a layout primitive. The question of how it pairs with Zod or React Hook Form — what the `error` → field id contract looks like, whether the DS exports a `useField` hook — is deferred. The answer should be driven by what the first real form-heavy consumer actually needs. Designing the contract speculatively risks building the wrong abstraction.

**Locale-aware date and number tokens.** `Stat` displays numerals; `Footer` displays a copyright year. Neither requires locale-aware formatting at the DS level today. If a consuming surface targets multiple locales, this becomes relevant — but it is the surface's responsibility until a clear pattern emerges across two or more consumers.

**CSS-in-JS variant of tokens.** Some consumers cannot ship `tokens.css` as a static file (e.g. they use a strict CSP without `<link>` tags, or they need runtime token overrides). A JS-imported token object — `import tokens from '@poukai-inc/ui/tokens.js'` — would serve them. This is low priority: no current consumer has this constraint, and implementing it risks fragmenting the token contract (one source of truth becomes two).
